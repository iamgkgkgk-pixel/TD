## 使用说明

### 标记系统
- `⭐` = 核心里程碑交互，必须重点把关
- `⬜` = 待执行
- `🔄` = 进行中
- `✅` = 已完成并验收通过
- `🚧` = 质量门控（阶段验收点）
- `🔀` = 可并行执行的交互

### 上下文管理策略（⚠️ 关键！）

Claude没有跨对话长期记忆，随着交互推进上下文会丢失。请执行以下策略：

**策略1：项目文档喂入**
> 每次开启新的Claude对话时，将以下文件作为上下文附件喂入：
> - `GDD.md`（游戏设计文档）
> - 当前阶段的关键代码文件列表
> - `项目状态摘要.md`（每完成一个子阶段更新一次）

**策略2：检查点摘要**
> 每完成一个子章节（如2.2核心框架），让Claude执行：
> "帮我输出当前项目状态摘要：已完成的模块列表、关键设计决策、待解决问题、下一步计划。输出为Markdown格式。"
> 将结果保存为 `项目状态摘要.md`

**策略3：代码仓库作为记忆**
> 所有Claude输出的代码、文档、配置表都及时入库（Git），下次交互直接附带相关文件即可恢复上下文。

### 输入格式模板

```
[阶段X - 第N条]

背景：（上一条的结果/当前项目状态摘要）
上下文文件：（附带的GDD/代码/配置等文件列表）
需求：（具体要Claude做什么）
约束：（技术限制/风格要求/性能要求）
产出格式：（代码/文档/配置表/设计稿）
验收标准：（怎么判断这条交互的产出合格）
```

### 🤖 Prompt质量修饰符（v3.0新增——将Claude产出质量提升一个档次）

在每条Prompt末尾，根据产出类型追加以下修饰符：

| 产出类型 | 推荐追加的修饰符 |
|---------|----------------|
| C#代码 | `"给出完整的、可编译的C#代码，含中文注释，不要省略任何部分。"` |
| 设计文档 | `"用Markdown表格对比方案优劣，给出明确推荐和理由。"` |
| 数值表 | `"输出CSV格式，第一行是表头，可直接导入Excel。"` |
| 架构设计 | `"先给出整体架构图（用ASCII或Mermaid），再逐模块详细说明。"` |
| Bug修复 | `"先分析根因，再给出修复代码，标注修改了哪几行。"` |
| Review | `"按严重程度排序（P0→P2），每条给出问题、影响、修复建议。"` |
| 配置JSON | `"输出完整的JSON文件内容，可直接保存为.json文件。"` |

### 🎭 AI角色扮演审查模式（v3.0新增——让Claude做决策者而非执行者）

在每个阶段的质量门控前，追加以下角色扮演交互：

```
"你现在扮演一个月流水百万的微信小游戏制作人。
审查我的[GDD/代码/数值表/UI设计]，指出3个最可能导致项目失败的问题，
并给出你的修改建议。要求严格、犀利、不留情面。"
```

```
"假设你是目标用户（30岁男性，轻度手游玩家，通勤时间玩游戏）。
你玩了我的游戏5分钟后：
1. 什么最吸引你？
2. 什么最可能让你关掉游戏？
3. 什么会让你愿意花6元买首充？"
```


### 迭代流程

```
发送Prompt（附带上下文文件）→ Claude产出 → Check效果
    ├─ 通过验收标准 → 入库(Git) → 更新项目状态摘要 → 进入下一条
    ├─ 部分通过 → 追加修改Prompt → Claude重新产出
    └─ 方向有误 → 回退重做（参考Git历史）
```
